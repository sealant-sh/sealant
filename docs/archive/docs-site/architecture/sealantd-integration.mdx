`sealantd` is the in-sandbox daemon that exposes a control socket for executing commands, streaming
binary-safe IO, and emitting telemetry. This page describes how it is baked into sandbox images by
the build pipeline and how the orchestrator (control plane) reaches it through a typed Effect
service. The integration runs entirely over the real Docker runtime path: a sandbox built by the
production pipeline boots `sealantd`, and the orchestrator drives it — `exec`, binary-safe stdout,
and streamed telemetry — over a real transport.

<Callout type="info">
  The proof bar for this integration is concrete: a sandbox built by the real buildkit pipeline
  boots `sealantd`, and the orchestrator drives it through a typed Effect service over a real
  `docker exec` transport. That bar is met end-to-end, with every phase verified by breaking each
  test to confirm it is not vacuous.
</Callout>

## Architecture

Sandboxes run as Docker containers via the Docker runtime adapter at
`packages/sandboxes/src/runtime/docker-runtime-adapter.ts`. `sealantd` runs _inside_ the container,
listening on a Unix socket at `/run/sealant/control.sock`. The orchestrator reaches it from outside
the container — today via `docker exec`, later via the SSH gateway as a dedicated tunnel channel.

```text
orchestrator (Effect)                          sandbox container (docker)
┌────────────────────────────┐                 ┌────────────────────────────────┐
│ SealantRuntime service     │  docker exec    │ entrypoint:                    │
│  ├─ transport (exec/socat) │◄──────stdio────►│   sealantd --socket … --ws … & │
│  ├─ @sealant/runtime-client│   ⇆ unix sock   │   (then user workload)         │
│  │    SealantClient.fromStream(duplex)        │   /run/sealant/control.sock    │
│  └─ Effect Stream<Event>   │                 │   baked: COPY --from ghcr …    │
└────────────────────────────┘                 └────────────────────────────────┘
```

The Kubernetes / k3s adapters are stubs and are not on this path. The Effect service layer is
deliberately adapter-agnostic, so the SSH-gateway and Kubernetes transports can slot in later
without changing the consumer code.

## Integration layers

The integration is decomposed into stacked layers, from baking the binary into the image up to a
worker-facing consumer call.

| Layer               | What                                                                                                      | Where                                                            |
| ------------------- | --------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| L1 — Bake binary    | `COPY --from=ghcr.io/get-sealant/sealantd` plus `socat` for the relay, gated by a flag                    | `buildkit-builder.ts` `renderContainerfile`, distro `packageMap` |
| L2 — Launch         | Entrypoint starts `sealantd` behind a flag, socket `/run/sealant/control.sock`, workspace = workdir       | `buildkit-builder.ts` `renderSandboxEntrypoint`                  |
| L3 — Transport      | `docker exec -i <c> socat - UNIX-CONNECT:…` bridged to a Duplex stream (`-i` is load-bearing, never `-t`) | `packages/sandboxes/src/sealantd/transport.ts`                   |
| L4 — SDK            | `SealantClient.fromStream(duplex)` — generic Duplex transport                                             | sealantd `packages/runtime-client/src/client.ts`                 |
| L5 — Effect service | `SealantRuntime`: `exec`, `events: Stream`, scoped connection, typed results                              | `packages/sandboxes/src/sealantd/runtime.ts`                     |
| L6 — Consumer       | A worker-facing `execInSandbox` call plus an adapter-driven path                                          | `packages/sandboxes` + `apps/worker`                             |
| L7 — Telemetry sink | Forward the event stream to a message bus or persist it                                                   | `packages/rabbitmq` / db                                         |

## How a request flows

<Steps>

<Step>
### Bake (L1)

When the build flag is set, the buildkit pipeline adds
`COPY --from=ghcr.io/get-sealant/sealantd:<version>` to the Containerfile and installs `socat` into
the image. No base image (Fedora, Arch, NixOS) ships `socat` or an nc-with-unix variant, so the
relay must be added explicitly. The image is pulled from a public, multi-arch (amd64/arm64) GHCR
image rather than bundled into the build context — this is the production path and needs no
build-context plumbing.

</Step>

<Step>
### Launch (L2)

The sandbox entrypoint starts `sealantd --socket /run/sealant/control.sock --workspace … &` before
handing off to the user workload. Launch is gated behind a default-off environment flag, so existing
sandboxes are unaffected unless the flag is set.

</Step>

<Step>
### Connect (L3 + L4)

The orchestrator opens a transport:
`docker exec -i <container> socat - UNIX-CONNECT:/run/sealant/control.sock`. The `-i` flag is
load-bearing for binary-safe IO; `-t` (a TTY) must never be used. The resulting pipe is a byte-level
Duplex stream, which is handed to `SealantClient.fromStream(duplex)`. Because the client runs over
any byte stream rather than only a local socket, the same SDK path works over a `docker exec` pipe
today and a gateway tunnel later.

</Step>

<Step>
### Drive (L5)

The `SealantRuntime` Effect service wraps the client with a scoped `connect`, an `exec` operation, a
telemetry `Stream`, and tagged errors. The transport itself is a pluggable `SealantTransport` seam —
`docker exec` now, SSH gateway or Kubernetes later — so the service is adapter-agnostic.

</Step>

<Step>
### Consume (L6)

`execInSandbox(target, cmd)` is the worker-facing call, returning an `Effect` that resolves to
`{ stdout, exitCode }`. Targets are produced from a Docker container id or a runtime instance via
`sealantTargetForDockerContainer` / `sealantTargetForRuntimeInstance`. An end-to-end path drives the
real `DockerRuntimeAdapter.launch()` to obtain a container id, builds a target, and execs against a
live container through `SealantRuntime`.

</Step>

</Steps>

## Key decisions and tradeoffs

<Cards>
  <Card title="Default-off flag">
    `sealantd` is fully opt-in per build via the `enableSealantd` customization flag. With the flag
    disabled the buildkit pipeline renders a byte-identical Containerfile, giving zero blast radius
    for existing sandboxes.
  </Card>
  <Card title="docker exec + socat transport">
    The current transport is real and is what the worker can already do. The SSH-gateway tunnel is
    the intended production transport and is deferred — reaching an in-pod Unix socket needs a new
    tunnel channel.
  </Card>
  <Card title="GHCR image, not a bundled binary">
    The daemon is pulled with `COPY --from` from a public, multi-arch, verified-pullable GHCR image.
    This is the production path with no build-context plumbing. An offline fallback binary is kept
    under `.sealantd-work/sealantd`.
  </Card>
  <Card title="peer-cred enforcement">
    `sealantd` validates `SO_PEERCRED` (same uid, root, or an allowlist). Root (uid 0) is always
    allowed, and both the container and `docker exec` run as root, so no `--user` is needed.
    Enforcement is proven: an unauthorized uid is rejected even with the socket made world-writable.
  </Card>
</Cards>

<Callout type="warn">
  A peer-cred mismatch is the most likely failure mode if uids drift. If `docker exec` runs under a
  uid that differs from the one `sealantd` runs as, the connection is rejected. The fix is to align
  the uids, pass `--allowed-peer-uids`, or run both as the sandbox user.
</Callout>

## Telemetry

`SealantRuntime` exposes the daemon's events as an Effect `Stream`. The telemetry sink (L7) is the
natural place to forward that stream to a message bus or persist it for an execution record. This
layer follows the same shape as the existing RabbitMQ and database integrations, so an event
consumer can attach without reshaping the runtime service.

<Callout type="idea">
  The event stream is the seam for persisting an execution record. Forwarding `SealantRuntime`
  events to RabbitMQ or a store gives you a durable, queryable history of every command run inside a
  sandbox.
</Callout>

## What ran end-to-end

The integration was built and verified phase by phase against the real Docker runtime.

<Accordions>
  <Accordion title="P1 — SDK transport over any byte stream">
    `SealantClient.fromStream(Duplex)` lets the client run over any byte stream (such as a `docker exec` pipe) rather than only a local socket. The connect and spawn paths remain intact, verified by the SDK end-to-end suite.
  </Accordion>
  <Accordion title="P2 — Bake and launch">
    A default-off `enableSealantd` build flag (a customization-schema field, auto-threaded through the build plan's customization). When set, the buildkit pipeline adds the `COPY --from` GHCR layer, installs `socat`, and launches `sealantd --socket /run/sealant/control.sock --workspace … &` in the entrypoint. The disabled path renders a byte-identical Containerfile, so existing sandboxes are unaffected.
  </Accordion>
  <Accordion title="P3 — Real image and live proof">
    A standalone build script (`compileSandboxBuildSpec`, no worker or database) produces a real Fedora image with `sealantd` baked in. An end-to-end test boots it and bridges `docker exec -i … socat - UNIX-CONNECT:…` into `SealantClient.fromStream`: a `health` check returns HEALTHY, then `exec('/bin/echo hi')` yields `processStarted`, STDOUT `"hi\n"`, and `exitCode 0` over the real transport.
  </Accordion>
  <Accordion title="P5 — Effect service">
    The `SealantRuntime` Effect service: scoped `connect`, `exec`, a telemetry `Stream`, tagged errors, and a pluggable `SealantTransport` seam (`docker exec` now; gateway and Kubernetes slot in unchanged). Verified both docker-free at the unit level and through the real service end-to-end.
  </Accordion>
  <Accordion title="P6 — Consumer seam">
    `sealantTargetForDockerContainer` / `sealantTargetForRuntimeInstance` plus `execInSandbox(target, cmd): Effect<{ stdout, exitCode }>` — the worker-facing call. An end-to-end test drives the real `DockerRuntimeAdapter.launch()`, derives a container id and target, and execs against a live container through `SealantRuntime`. The change is purely additive and default-off, with no production launch-path changes.
  </Accordion>
</Accordions>

## What is next

<Steps>

<Step>
### Release the SDK with `fromStream`

Tag and publish the daemon release that carries `fromStream`, then approve the release gate. The
branch swap to the published `@sealant/runtime-*` packages and the matching image pin are already
staged; once the release publishes, a fresh install plus a re-run of the end-to-end suite finishes
the swap.

</Step>

<Step>
### SSH-gateway socket tunnel

Build the production transport: a `socat` / `nc -U` channel in `apps/ssh-gateway` that replaces
`docker exec` for reaching the in-pod control socket.

</Step>

<Step>
### Pick the product surface

Decide how to expose the worker-facing `execInSandbox` — as an api-contracts operation plus route,
wired into the worker's live flow, and/or by recording control-socket availability on the runtime
instance. Telemetry persistence (forwarding the event stream to a message bus or store) belongs here
too.

</Step>

<Step>
### api-contracts alignment

Align the events and results at the API boundary with the Effect Schema api-contracts so the typed
surface matches what the runtime emits.

</Step>

<Step>
### Multi-arch validation and pin bump

Validate on amd64 (proven on arm64; the image is multi-arch) and advance the GHCR image pin,
alongside the telemetry persistence work above.

</Step>

</Steps>
