Before any product decision, we needed an honest picture of what Sealant already is — grounded in
the code of both repositories (`sealant-core` and `sealantd`), not the marketing. This study reads
the real source, labels every capability by its true maturity, and draws the hard boundaries that
the rest of the design process has to respect.

<Callout type="info">
  Sealant is a runtime for running and recording untrusted code — AI-agent or human — inside
  isolated workspaces, exposed through one stable contract. Its defining primitive is the
  **execution record**: every observed fact is minted at one deterministic sequencing point, carries
  correlation IDs and honest provenance, and is replayable as a pure fold of an append-only log.
  Sealant reports facts and does not judge.
</Callout>

## Method

The study was deliberately adversarial about its own claims. Seventeen parallel reader agents read
the real source across both repos, then two synthesis agents produced a capability/boundary map and
a developer-perspective feature brainstorm. Every claim below was verified against files, not
READMEs. Maturity labels are honest, drawn from one of four levels:

| Label         | Meaning                                    |
| ------------- | ------------------------------------------ |
| `working`     | Production-shaped and exercised end to end |
| `mixed`       | Real core, but with a partial or weak edge |
| `scaffold`    | Structure exists, behavior does not        |
| `placeholder` | Naming/positioning only; no code           |

## The thesis: what the platform really is

Sealant's true core is two layers that already exist and work.

<Cards>
  <Card title="sealantd — the Rust PID-1 daemon">
    Owns process/PTY execution, signal and reaping semantics, filesystem and network evidence
    collection, and a durable, crash-recoverable telemetry spool — all reachable over one
    length-prefixed Protobuf Unix-socket control protocol whose `.proto` is the single source of
    truth for codegen.
  </Card>
  <Card title="A TypeScript SDK seed">
    `@sealant/runtime-client` and `@sealant/runtime-protocol` (v0.4.0), generated from that same
    `.proto`, plus an Effect-native composition/orchestration layer in `sealant-core`: build,
    publish, launch on Docker, a RabbitMQ job queue, Postgres state, telemetry ingest, and an SSH
    gateway.
  </Card>
</Cards>

The platform's defining primitive is the execution record. Every observed fact — process lifecycle,
byte-exact I/O with stream offsets, file diffs, network egress metadata — is minted at one
deterministic sequencing point, carries correlation IDs and honest provenance (`captureMethod` plus
Observed/Inferred/Unknown confidence), and is replayable as a pure fold of an append-only log.
Sealant supplies the environment and the evidence, not the agent.

## The platform/products split already exists in the code

The code already draws a clean line, and that line is physically enforced: communication is IPC-only
(never FFI), and `runtime-client` is the only thing that knows the wire.

<Cards>
  <Card title="Platform">
    The `sealantd` daemon (Rust crates `sealant-protocol`, `sealant-control`, `sealant-process`, `sealant-pty`, `sealant-telemetry`, `sealant-eventlog`, `sealant-fs`, `sealant-network`, `sealant-runtime-core`, `sealantd`, `sealantctl`) plus the generated wire SDK (`@sealant/runtime-protocol`) and the thin typed client (`@sealant/runtime-client`). These live in a **separate repo** — `sealant-core` vendors no `.proto` and pins the SDK at `^0.4.0`. The platform runs code in a workspace, multiplexes byte channels, and emits a durable, correlated, provenance-honest execution record over one socket — nothing product-specific.
  </Card>
  <Card title="Products">
    Everything in `sealant-core` that consumes that SDK: the workspace composition/build/launch engine (`@sealant/workspaces`), the control-plane API plus RabbitMQ worker plus Postgres state (`apps/api`, `apps/worker`, `packages/{db,rabbitmq,validators,api-contracts}`), the telemetry read-model pipeline (`packages/telemetry`), the SSH gateway, the web and mobile consoles, source/issue/auth integrations, and the not-yet-built Verify/Repro/Handoff and issue-to-PR engines.
  </Card>
</Cards>

<Callout type="idea">
  The real split decision is **not** "extract a platform" — the platform already lives in its own
  repo. It is: what *additionally* graduates from products into the platform SDK? The ambiguous
  middle is `packages/telemetry`, the workspace blueprint schema, and the execution-record read
  models — generic enough to be SDK surface, but currently living in the product monorepo.
</Callout>

## Capability inventory

### Platform capabilities

These are the primitives products consume.

| Capability                                                | Maturity  | Notes                                                                                                                                                                                                                                                                                                                                                                             |
| --------------------------------------------------------- | --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Control protocol (`sealant-protocol` + `sealant-control`) | `working` | proto3 wire contract: 24 commands, 13 results, 12 event payloads, 18 error codes; 4-byte length-prefixed Unix-socket/stdio transport; byte-conduit multiplexing via `ChannelId`; `SO_PEERCRED` peer-uid auth; oversized-frame DoS defense; fuzz harness. One cross-language contract.                                                                                             |
| Process and PTY execution                                 | `working` | Spawns non-interactive processes plus interactive PTY sessions in their own process groups; dual-taps I/O as lossy-redacted telemetry **and** lossless backpressured byte channels; timeouts (SIGTERM then SIGKILL); signals/kills process groups; reaps orphans via subreaper; SFTP and TCP-forward conduits.                                                                    |
| Durable telemetry bus and event log                       | `mixed`   | One deterministic sequencing point; append-only, CRC32-checked, segmented on-disk spool with crash recovery, replay-on-restart, and disk-bound eviction; priority-aware backpressure preserves Critical events. The ack model is the weak link (see boundaries below).                                                                                                            |
| Filesystem and network evidence collectors                | `mixed`   | Unprivileged SHA-256 baseline/final snapshots plus an inotify live watcher (inferred renames, overflow recovery); a cooperative userspace HTTP/HTTPS-CONNECT egress proxy emitting per-request evidence; raw `direct-tcpip` forward. Privileged eBPF/netlink/payload modes are inert schema only.                                                                                 |
| Daemon composition and PID-1 supervisor (`sealantd boot`) | `working` | Boots the container, clones the repo with scoped short-lived creds, applies dotfiles, runs lifecycle steps, supervises the agent harness as a managed child, and wires all subsystems behind one socket; configured by a `SEALANT_*` env contract. Replaces the in-container sshd. Baked into every image as `ENTRYPOINT ["sealantd","boot"]`.                                    |
| TypeScript runtime SDK (`@sealant/runtime-client`)        | `working` | Connects to or spawns `sealantd`; drives exec/PTY/stdin/signal/shutdown; multiplexes byte channels (attach/forward/sftp) with SSH-style half-close; exposes `events()` as `AsyncIterable<EventEnvelope>`. The seed of the eventual public SDK. Roughly 10 of 24 wire commands still lack typed convenience methods.                                                               |
| Workspace composition engine (`@sealant/workspaces`)      | `mixed`   | `WorkspaceBlueprint` to BuildKit Containerfile render/compile, OCI registry publish (Zot), `docker run -d` launch (runc/runsc), runtime-adapter selection, GitHub-installation scoped auth, Repology package standardizer. The k8s/k3s adapters throw "not implemented yet."                                                                                                      |
| Control-plane API + worker + state                        | `working` | Contract-first Effect `HttpApi` (`POST /v1/workspaces`, OpenAPI emitted) to Postgres rows (blueprint, queued-attempt, build-job) to a RabbitMQ pointer to a background worker (lease-based at-least-once claim) that compiles/publishes/launches and ingests telemetry. Roughly 45 migrated Postgres tables (Drizzle).                                                            |
| SSH gateway (control-socket bridge)                       | `working` | One SSH entrypoint (`ws-<id>@gateway`) for all workspaces; pubkey-auth, resolve principal, the API authorizes principal × workspace, then bridges SSH channels to daemon control commands (shell to `openSession`, `direct-tcpip` to `openForward` for VS Code Remote, sftp to `openSftp`). Does not dial an inner sshd.                                                          |
| Telemetry read-model / run-inspector substrate            | `mixed`   | Event-sourced ingest and query over the daemon stream: gap/drop detection, content-addressed artifact offload, byte-exact scrollback reconstruction by `streamOffset`, projections that are pure re-folds (projection equals rebuild). The north-star folds (`getRunRollup`, `reconstructFileTree/ProcessTree/NetworkInFlight`, `tail`) throw `TelemetryQueryUnimplementedError`. |

### Product candidates

These are the things that could be "Built on Sealant."

| Candidate                                  | State today                                                                                                                                                                                                                                                |
| ------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Run Inspector / Execution Record viewer    | The capture and store side is production-shaped; the read side is unbuilt. `RunTelemetry` is exposed via no route, and the web diff/trace/validation pages hard-redirect to a raw spec dump.                                                               |
| Sealant Verify / Repro / Handoff           | Prominently marketed (`apps/marketing` lines 739–753), but zero code anywhere outside marketing. Positioning and naming placeholders today.                                                                                                                |
| Issue-to-PR workflow engine                | Full DB lineage plus issue import plus a kanban board UI exist, but zero orchestrator consumes `issueWorkflowExecution*`. The engine that runs the change and builds the PR is unbuilt.                                                                    |
| Managed workspace / remote dev environment | The most-complete end-to-end path today: real BuildKit compile, Zot publish, `docker run`, ssh-gateway with `direct-tcpip` to `openForward` plus working editor deep-links. Limited to single-host Docker, ephemeral-only.                                 |
| Agent-execution control plane / fluent SDK | A verified mismatch: the marketed `workspaces.create({repository, harness}).harness.run(prompt)` does not exist. The actual contract is REST-style and needs `{ownerUserId, registryId, repository, tag, spec, sourceSelection?}` with no `harness` field. |

## Hard boundaries — what Sealant is NOT

Drawn from the threat model and the code, these are the limits the product work must not paper over.

<Accordions>
  <Accordion title="Tamper-evident, not tamper-proof">
    Against the default deployment — a root workload sharing the daemon's UID/PID/net/mount namespaces — a workload can kill the daemon, read `/proc/<pid>/{environ,mem}`, connect to the `0600` socket as a same-UID peer, and rewrite the spool plus its CRC32 checksums. CRC32 is accidental-corruption detection, explicitly not cryptographic.
  </Accordion>
  <Accordion title="Not an authentication system">
    The socket boundary *is* the access boundary. `sealantd` does not authenticate end users; it trusts the already-authenticated caller (the SSH gateway holds auth). Authorization is coarse — any peer past the `0600` plus `SO_PEERCRED`/uid gate is fully trusted. Off Linux (macOS dev) the peer check is skipped entirely.
  </Accordion>
  <Accordion title="Not an enforced egress boundary">
    The egress proxy is cooperative, honoring only `HTTP_PROXY`. A workload can ignore the env, open raw sockets, or use pinned IPs and be invisible. There is no allow/deny/block/rate-limit — only evidence.
  </Accordion>
  <Accordion title="Not exactly-once telemetry">
    At-least-once by design; dedup is pushed to the SDK keyed on `EventId`. Worse than advertised: `high_water` advances and the spool is acked the instant `broadcast.send()` is called, regardless of whether any subscriber received it — "durable until first broadcast attempt," not "until SDK confirmed." No resume-from-sequence command exists.
  </Accordion>
  <Accordion title="Not a terminal emulator, model, image builder, scheduler, or runtime">
    It is not a terminal emulator, an AI model/agent, an image builder, a Kubernetes scheduler, an SSH auth server, or a container runtime. It forwards raw PTY bytes without interpreting them.
  </Accordion>
  <Accordion title="Not multi-runtime, not persistent">
    Only the Docker adapter is real; k8s/k3s throw. The Docker adapter refuses anything but ephemeral. No stop/terminate/TTL/GC of running containers exists.
  </Accordion>
  <Accordion title="Not a connection-resilient control channel">
    Channels and event broadcast are connection-scoped: a dropped connection kills all PTY attaches, forwards, and sftp bridges plus pending requests. The TS client only retries the initial connect.
  </Accordion>
  <Accordion title="Not an encryption/KMS boundary">
    `secret_versions` stores ciphertext plus key-id plus sha256, but the DB layer performs no crypto; runtime secrets are plaintext `-e` env to `docker run` (visible via `docker inspect`); redaction is heuristic/prefix-based (10 token shapes, 16-byte minimum).
  </Accordion>
  <Accordion title="Not multi-tenant at the data layer">
    Ownership is per-user only — there is no org/team entity. The control-plane API has no end-user auth middleware; `ownerUserId` is a trusted request input.
  </Accordion>
  <Accordion title="Not the judge">
    No confidence scores, "safe to merge," risk dashboards, or verdicts.
  </Accordion>
</Accordions>

## Maturity summary

A condensed read of real versus scaffold across the whole system.

<Cards>
  <Card title="Production / working">
    The `sealantd` daemon core (control protocol, framing, peer auth, process/PTY execution,
    signals/reaping, composition/PID-1 boot); the TS runtime SDK seed; the SSH gateway; the RabbitMQ
    worker plus Postgres state model (roughly 45 tables, lease-based at-least-once claim); the
    control-plane API (`POST /v1/workspaces` to queued build); the telemetry ingest/store pipeline.
    The most-complete end-to-end path is create-blueprint, build, publish, docker-launch, SSH/editor
    access.
  </Card>
  <Card title="Working but partial">
    Telemetry durability (the spool exists, but the ack advances on broadcast-attempt not receipt,
    and `fsync` is hardcoded `Never` with a 1s loss window); filesystem evidence (snapshots and
    watcher work, but there are no diff artifacts, no FS redaction, and kill switches are inert);
    the web console (real workspace lifecycle, but Repositories/Profiles/Registry are mock, and
    diff/trace/validation redirect).
  </Card>
  <Card title="Scaffold / placeholder">
    Privileged network modes (eBPF/netlink/payload — inert schema, no collector process); pidfd
    race-free signaling (kernel detection only, still uses `killpg`); the k8s/k3s adapters (throw);
    the issue-to-PR execution engine (full DB lineage plus board UI, zero orchestrator); the
    telemetry read-model API (no route, folds throw); Verify/Repro/Handoff (zero code); the mobile
    app (read-only live, write paths mock); the Electron app (empty scaffold).
  </Card>
</Cards>

## Tensions to resolve

These are the open questions the study surfaced — the agenda the rest of the design process has to
settle.

<Steps>

<Step>

### Narrative split

Two product stories coexist. The rebranded marketing page (the doc-declared "canonical reference")
sells a harness-neutral "Runtime for AI developer agents" — platform plus SDK plus
Verify/Repro/Handoff. The in-repo README still describes "fast isolated workspaces plus issue-to-PR
workflows" as the two product loops, with a "workspace plus issue workflow" terminology contract.
Marketing has quietly demoted "issue workflow" to a product (Handoff).

</Step>

<Step>

### SDK overclaim (verified)

The marketing hero's fluent `workspaces.create({harness}).harness.run()` returning
`{result, changes, artifacts, record}` does not match the real REST contract (no `harness` field).
Either the fluent SDK ships from the external runtime-client repo and must be made real and public,
or the hero is fiction. This is the single most important platform/SDK shape decision.

</Step>

<Step>

### Open-source / platform location

The footer claims an "Open-source runtime," but the `.proto` and SDK live in an external repo. The
platform we want to split out already lives elsewhere; `sealant-core` is the products monorepo.

</Step>

<Step>

### Execution record: captured but not consumable

The central marketed primitive is genuinely captured and stored, but exposed via no API route, the
inspector pages redirect to JSON, and the north-star folds throw `Unimplemented`. The platform
records facts that no product can yet read.

</Step>

<Step>

### Flagship loop unbuilt

README's "issue-to-PR" and marketing's "Handoff" are the same unbuilt thing under two names — rich
DB lineage, a board UI, but no execution engine.

</Step>

<Step>

### Durability and security: marketing vs code

"Authoritative, replayable execution record" plus "secret redaction" plus "network visibility as
controls" versus the reality of tamper-evident-only, broadcast-attempt ack, heuristic redaction, and
a bypassable proxy. Audit-grade framing requires a privilege boundary that is only a
"Should"-priority, possibly-unmet requirement.

</Step>

<Step>

### Doc drift everywhere

README understates maturity; `integration-brief.md` still says "every crate is a one-line stub"
(false); ssh-gateway docs describe the removed inner-sshd design; `globals.css` header describes an
old flat design that contradicts its own v3 tokens; the blueprint carries a dead `enableSealantd`
flag though sealantd is now mandatory PID-1.

</Step>

<Step>

### Versioning unfinished for a platform that ships external SDKs

Only `schema_version 1` exists; exact-match negotiation means a v2 daemon hard-rejects all v1
clients; there is no buf lint/breaking-change detection; `RequestId` is documented as an idempotency
key but no dedup exists.

</Step>

</Steps>
