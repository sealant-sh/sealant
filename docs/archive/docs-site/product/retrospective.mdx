This is a factual and advisory look across the `sealantd` daemon and the `sealant-core` monorepo,
written after the `sealantd` integration landed. The first three sections describe how things work
today — image building, the SSH proxy versus the daemon, and where captured telemetry goes. The last
three are recommendations: cleaning up image building, sorting out the SSH gateway, and whether a
network MITM proxy is warranted. File references are load-bearing and cross-checked against the
source, not the docs.

<Callout type="info">
  Three findings, in priority order: the telemetry capture pipeline is fully built but **dead-ends
  in memory** in production; the SSH gateway accepts **any upstream host key** today (exploitable
  now); and workspace orchestration is generated as **~370 lines of string-concatenated bash** with
  no real init system. The unifying theme of every recommendation is that the daemon already
  implements the hard parts — the work is to route through it rather than around it.
</Callout>

<Callout type="warn">
  **Scope note.** "Today" reflects the source as committed: the `sealantd` binary is baked from
  `ghcr.io/get-sealant/sealantd:0.2.0`. A stale test still asserts `0.1.2`, but the source is the
  current state. `sealantd` is **opt-in and default-off**; the disabled path renders byte-identical
  to the pre-`sealantd` output by design.
</Callout>

## How Sealant builds images now

A 1,255-line TypeScript file renders a `Containerfile` and an `entrypoint.sh` as arrays of raw
strings, then runs `docker build` plus `docker save` to a local OCI tarball; a separate registry
client loads, tags and pushes it. There is no template file and no init system — the entrypoint _is_
the orchestration.

Two files drive every build: the renderer/executor
`packages/workspaces/src/buildkit/buildkit-builder.ts` and the worker-side driver
`packages/workspaces/src/worker/process-workspace-build-job.ts`.

### The end-to-end flow

The worker entry point is `processWorkspaceBuildJob` (`process-workspace-build-job.ts:102`):

<Steps>
  <Step>**Claim and parse.** Claim a job, then parse the blueprint with `newWorkspaceSchema`.</Step>
  <Step>**Compile.** Call `compileWorkspaceBuildSpec` (the call into `buildkit-builder`).</Step>
  <Step>
    **Select the artifact.** Pick the publishable `oci-image` artifact (`:153`), which must have
    `loader === "docker-load"`.
  </Step>
  <Step>
    **Publish.** Push through `registryClient.publishOciImage`, then persist `publishedReference`
    and `Digest`.
  </Step>
  <Step>
    **Launch.** Upsert the runtime instance, resolve clone/dotfiles auth, and launch via a selected
    runtime adapter.
  </Step>
</Steps>

A `buildSucceeded` guard (`:139`, `:254`) ensures a post-build failure never re-marks the build
itself failed.

Inside `compileWorkspaceBuildSpec` (`buildkit-builder.ts:1193`): parse input, select the OS family,
run `mapBlueprintToResolvedImagePlan`, then `writeBuildContext` (temp dir), then `buildImageTarball`
(the actual `docker build` plus `docker save`), and return a validated result carrying one
`oci-image` tarball plus two `metadata` JSON artifacts.

### What is rendered, and how

- **OS family.** `auto` tries `["fedora","arch","nix"]` in order (`:386`); support checks reject
  version-pinned packages, unknown harnesses, and any non-dotfiles runtime input
  (`getBuildkitSupportForOs :333`).
- **The Containerfile** (`renderContainerfile`, `:1079`) is an array of strings joined with `\n` —
  `# syntax=docker/dockerfile:1.7`, a `FROM` on the base image, a single cache-mounted package `RUN`
  per distro (dnf / pacman / nix), the harness install, shell config, the entrypoint COPY/chmod,
  then (only when enabled) the `sealantd` bake.
- **The entrypoint** (`renderWorkspaceEntrypoint`, `:687–982`) is also a string array — roughly 296
  lines of bash that decodes SSH keys, optionally rewrites `/etc/passwd|group|shadow` and launches
  an in-container `sshd`, clones the repo, applies dotfiles, runs lifecycle steps, backgrounds
  `sealantd`, then `exec`s the harness.
- **Per-distro catalog.** `distroDefinitions` (`:163`) pins `fedora:41` / `archlinux:latest` /
  `nixos/nix:latest` with their package managers, internal packages, shell paths and `sshdPath`,
  plus a symbolic-id `packageMap`. `socat` is added only when `sealantd` is enabled (`:587`).

### Build, tag, push and cache

`buildImageTarball` (`:1165`) always sets `spec.push = false` — it only builds and `docker save`s a
local tarball (arch is pinned to `linux/amd64`). Pushing is a separate step in
`ZotRegistryClient.publishOciImage` (`registry/client.ts:324`): `docker load`, then `docker tag`,
then `docker push`, then resolve the digest via `headManifest`.

Caching relies entirely on BuildKit's default layer plus cache-mount behaviour (`DOCKER_BUILDKIT=1`,
dnf/pacman cache mounts); there are **no** `--cache-from` / `--cache-to` flags and **no** registry
digest check to skip an unchanged build.

## The SSH proxy and the daemon — how they relate today

They are **two fully independent paths that never meet**. The gateway speaks SSH to the container's
`sshd` over TCP; the daemon is reached by `docker exec` plus `socat` to a Unix socket speaking
length-prefixed protobuf. Zero code references connect them — proven by a passing e2e where `sshd`
is not even running yet the daemon is fully reachable.

Inside a workspace these are two separate processes the entrypoint starts independently. They differ
in every dimension:

|               | SSH gateway path                              | Daemon control path                                                    |
| ------------- | --------------------------------------------- | ---------------------------------------------------------------------- |
| **Listener**  | OpenSSH `sshd` on a TCP port (default 2222)   | Unix socket `/run/sealant/control.sock`, mode `0600`                   |
| **Protocol**  | SSH (KEX, channels, auth)                     | Binary length-prefixed protobuf control frames                         |
| **Transport** | Network SSH, multi-hop relay                  | `docker exec -i <ctr> socat - UNIX-CONNECT:<sock>`                     |
| **Auth**      | Publickey allowlist at the gateway boundary   | `SO_PEERCRED` uid check, fails closed                                  |
| **Carries**   | Interactive shell / exec / SFTP / TCP-forward | health / capabilities / exec / writeStdin / signals / telemetry events |

### What the gateway does

`apps/ssh-gateway/src/gateway-server.ts` is a multi-hop relay: one client SSH connection maps to one
workspace and one lazily-opened upstream SSH connection, with channels multiplexed over it. It auths
publickey-only against a static allowlist, parses the `workspaceId` from the SSH username
(`ws-<id>`), resolves the target via `GET /v1/workspaces/{id}/ssh-target` on the core API, then
opens an upstream `ssh2.Client` to the **container's `sshd`** (`connectToUpstream :65`). It forwards
`shell` / `exec` / `subsystem` (SFTP) / `tcpip` plus PTY/window-change/signal metadata so the
terminal behaves transparently — including `forwardOut` through the upstream session to make VS Code
Remote dynamic forwarding work.

### Why the daemon is not reached via the gateway

The daemon serves a `tokio::net::UnixListener` (`sealant-control/src/server.rs:206`) with
`SO_PEERCRED` uid authorization (`peer.rs`); it has **no SSH listener at all**. The only implemented
transport on the `sealant-core` side is the docker-exec/socat bridge: `SealantTarget` is a
discriminated union whose only variant today is `{ kind: "docker-exec" }`
(`sealantd/runtime.ts:44`), and `dockerExecTransport.open` spawns the `socat` child and adapts its
stdio into a `Duplex` carrying protobuf frames. k8s/k3s targets return `undefined` — only Docker is
bridgeable (`sealantd/target.ts:47`).

<Callout type="info">
  **Direct proof of independence.** The `sealantd` e2e fixture exercises the full docker-exec control path with `ssh: { enabled: false }` (`target.e2e.ts:80`) — the `sshd` the gateway would target is not even running, yet `sealantd` is fully reachable. The only cross-mentions in either codebase are forward-looking comments describing a *hypothetical* future SSH-gateway transport (`runtime.ts:15,41`) — no code path, config, or shared identifier connects them today.
</Callout>

## What happens to the data the daemon captures

The capture, redaction, envelope/sequence, durable-spool and Effect-Stream plumbing are all built —
but the pipeline **dead-ends in memory**. As deployed the bus is the non-durable Direct mode (no
`--spool-dir`), events reach only live-connected clients, and the one consumer keeps stdout plus
exit code and discards everything else. Nothing is persisted to a DB, forwarded to a queue, or
written to disk.

### What is captured

All telemetry is one typed enum, `EventPayload` (`sealant-protocol/src/event.rs`), wrapped in an
`EventEnvelope` that the bus stamps with `eventId`, `runtimeId`, monotonic `sequence`, timestamps,
`captureMethod` and `confidence`. Twelve event types span four subsystems:

| Subsystem        | Events                                                                               | Source                                                         |
| ---------------- | ------------------------------------------------------------------------------------ | -------------------------------------------------------------- |
| Process / PTY IO | `process.started`, `process.exited`, `io.chunk` (base64, binary-safe, redacted)      | `sealant-process/src/runtime.rs`, `sealant-pty/src/session.rs` |
| Filesystem       | `file.changed`, `file.snapshotCompleted`, `file.diffAvailable`, `file.watchOverflow` | `sealant-fs/src/runtime.rs`, `watcher.rs` (snapshot + inotify) |
| Network          | `network.request`, `network.sourceObserved`                                          | `sealant-network/src/proxy.rs` (egress proxy)                  |
| Runtime / bus    | `runtime.stateChanged`, `runtime.heartbeat`, `telemetry.dropped`                     | lifecycle + the bus itself                                     |

### On-daemon retention

The bus (`sealant-telemetry/src/bus.rs`) has two modes: **Direct** (broadcast only, no persistence)
and **Durable** (queue, then WAL spool, then broadcast, then ack). `build_bus`
(`sealantd/src/runtime.rs:64`) picks Durable _only_ when `--spool-dir` is set — and the production
launch command bakes only `--socket` and `--workspace`, so **Direct, non-durable mode runs in
practice** and nothing touches disk. Even in Durable mode the spool
(`sealant-eventlog/src/spool.rs`) is a crash-recovery WAL, not a queryable store: records are
deleted on ack or evicted under a disk bound. Under pressure, `Low` / `Normal` events are dropped
and counted while `Critical` events spill inline.

### What sealant-core does with the events

<Callout type="warn">
  **Finding: captured and exposed, but not persisted or forwarded.** The pipeline builds up to a
  live in-memory stream and stops there.
</Callout>

The control socket forwards each `EventEnvelope` as `ServerMessage::Event` to **currently-connected
clients only** (`server.rs:96`). On the consumer side: the SDK client buffers into an in-memory
`#eventQueue` drained by an async iterator, dropping un-pulled events on close
(`runtime-client/src/client.ts:298`); `SealantRuntime.events` is just an Effect `Stream` wrapper the
service never consumes; and `execInWorkspace` — the **only** code that consumes the stream, and a
**test-only** caller — concatenates STDOUT `io.chunk` bytes plus reads `exitCode`, then discards
stderr, file, and network events when the scope closes (`sealantd/target.ts:102`). There is **no**
telemetry path through `packages/db` or `packages/rabbitmq`.

```text
CAPTURE (process/pty pipes, fs snapshot+inotify, egress proxy)
  → EventBus.publish  (stamp id/sequence/time, redact secrets, assign priority)
  → Direct mode broadcast channel        [as deployed — no disk]
  → control socket → live-connected clients only (fan-out; disconnected clients miss events)
  → SDK #eventQueue → events() iterator  (un-pulled events dropped on close)
  → execInWorkspace (test-only)  keeps STDOUT + exitCode, DISCARDS the rest on scope close
  → DEAD END: nothing in packages/db, nothing in packages/rabbitmq, nothing on disk in prod
```

## Recommendation: cleaning up image building

Stop generating bash by string concatenation. Move the workspace orchestration into `sealantd` as a
`sealantd boot` supervisor subcommand (it already _is_ a child-subreaper with a process registry),
and shrink the Containerfile to thin per-distro templates. TypeScript keeps the one job it is good
at: planning.

### The pain

`renderWorkspaceEntrypoint` is roughly 296 lines of string-concatenated bash, with a second roughly
80-line copy of the dotfiles logic in `renderDotfilesStep`. Correctness rides on getting array
boundaries, nested heredocs (bash generating bash across two template layers), and `\n` escaping
right by eye. Real security-sensitive work — decoding auth keys, writing askpass shims, rewriting
`/etc/shadow` under a read-only rootfs, host-key generation, dotfiles auto-detection — lives in the
least testable language available. There is **no init system**: `sshd` and `sealantd` are
backgrounded with `&` plus a hand-rolled `trap`, so nothing is PID 1 in the supervisor sense (no
zombie reaping, partial signal propagation). The only test is snapshot assertions over generated
text.

### Options

The decision spans three axes — the Containerfile, the init/supervision strategy, and where the
orchestration logic lives.

<Callout type="idea">
  **Verdict: A1 → (later A2) + B3 + C2.** Move orchestration into a `sealantd boot` supervisor
  subcommand and reduce the Containerfile to thin per-distro templates. You already own a process
  supervisor — `sealant-process/src/platform.rs` implements subreaper, orphan reaping, `pidfd` and
  `no_new_privs`; the bash `&` + `trap` is a worse reinvention of it, and the topology is inverted
  (the harness is PID 1's `exec` target while `sealantd` is a backgrounded child). The correct shape
  is **sealantd as PID 1, harness + sshd as supervised children**. Plain `tini` leaves the actual
  complaint in bash; `s6-overlay` trades bash-templating for s6-dir-templating. C1 (bash +
  shellcheck + bats) is the right fallback *only* if a Rust-in-image binary is rejected.
</Callout>

### Phased plan

No broken workspaces at any step.

<Steps>
<Step>
**Phase 0 — Lock behaviour.** Add `shellcheck` over the rendered entrypoint in CI and pin the current snapshot; this is the safety net for every later phase.
</Step>
<Step>

</Step>
<Step>
**Phase 2 — Introduce `sealantd boot` (the core move, behind a flag).** A `clap` subcommand that becomes subreaper, reads a typed `BootConfig` JSON, performs clone/auth/sshd/dotfiles/lifecycle, then supervises the harness plus control server. Gate it: emit `ENTRYPOINT ["sealantd","boot",…]` only on opt-in; legacy bash stays byte-identical. Existing `*.e2e.ts` specs become the parity acceptance test.
</Step>
<Step>
**Phase 3 — Flip the default, keep a bash fallback** for one release and for the `enableSealantd === false` path.
</Step>
<Step>

</Step>
</Steps>

## Recommendation: sorting out the SSH gateway

Consolidate onto the daemon: keep SSH termination plus auth in the gateway (per ADR-0004) but
replace the upstream `ssh2.Client` with a control-protocol client to `/run/sealantd.sock`, deleting
the in-container `sshd` and its second crypto hop. Fix the host-key trust bug _now_. Rewrite in Rust
**last**, not first.

### Current weaknesses

The gateway is a Node plus `ssh2` multi-hop relay — two full SSH stacks per session (double crypto,
double hop), with an entire second `sshd` baked into every image purely to terminate the inner hop.
The worst issue: upstream host-key handling defaults to accepting **any** key
(`gateway-server.ts:106–117`) — even "strict" mode has no pinning, no known_hosts, no TOFU. Auth is
coarse (one allowlist, one gateway key opens **every** workspace; no per-principal to per-workspace
authorization), there is no rate-limiting or per-connection caps, and observability is ad-hoc
`console.error` lines.

### The consolidation — and the gaps that block a naive tunnel

The gateway's SSH verbs map 1:1 onto control-protocol commands that already exist (`command.rs`):
`shell` → `openSession`, `exec` → `exec`, `window-change` → `resizePty`, `signal` → `signalProcess`,
`close` → `closeSession`. Three real gaps remain:

1. PTY output leaves `sealantd` as **fire-and-forget broadcast telemetry** (`IoChunk` on a fan-out
   receiver, subject to lag/drop), not a reliable backpressured per-session stream.
2. **No SFTP subsystem and no arbitrary TCP `forwardOut`** (the egress proxy is HTTP/CONNECT only) —
   VS Code Remote dynamic forwarding has no daemon equivalent.
3. The SSH protocol still has to be terminated somewhere.

| Shape                                                                | What it is                                                                                                                                | Assessment                                                                                                       |
| -------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **A — gateway terminates SSH, tunnels control to the daemon socket** | gateway keeps auth + host-key + channels; upstream becomes a control-protocol client to `/run/sealantd.sock`; in-container `sshd` deleted | Recommended — keeps the auth boundary where ADR-0004 puts it; kills the second hop; reuses the built session API |
| B — daemon terminates SSH; gateway becomes a thin router             | `sealantd` speaks SSH on a port; gateway is an SNI/TCP router                                                                             | Relitigates a settled ADR; moves end-user auth into the daemon; widens in-container attack surface               |

<Callout type="idea">
  **Verdict: consolidate on shape A. On the explicit Rust question — eventually yes, but not now and
  not first.** The gateway's problems are roughly 80% design (double hop, host-key trust, auth
  granularity, observability) and 20% language. Rewriting in Rust (`russh` plus the
  `sealant-protocol` crate) *before* consolidation just reproduces the flawed double-hop shape in a
  new language. The type-sharing win is real **only** in shape A and only for the control-client
  half — the SSH-server half shares nothing with `sealantd` regardless of language. So: fix the
  design in Node first, adopt Rust as the *endpoint* of the migration. The accept-any-host-key fix
  must **not** wait for the rewrite — it is exploitable now.
</Callout>

### Phased plan

<Steps>
  <Step>
    **Phase 0 — Stop the bleeding (Node, days).** Default strict host-key checking on; implement
    real upstream host-key **pinning** (TOFU or key delivered via the target lookup); add structured
    tracing with connection-to-workspace-to-session correlation; add auth-attempt rate-limiting and
    a cap on `forwardOut` channels.
  </Step>
  <Step>
    **Phase 1 — Daemon readiness (Rust).** Add a reliable, backpressured per-session output stream
    (an interactive mode that opts out of broadcast drop); add `forwardTcp` / `openForward` and an
    SFTP-equivalent — the two real gaps versus today's relay.
  </Step>
  <Step>
    **Phase 2 — Consolidate transport.** Tunnel the gateway to `/run/sealantd.sock`; rewrite
    `connectToUpstream` to open a control session instead of an `ssh2.Client`; **delete the
    in-container `sshd`** plus `workspace-ssh-shell`; add per-principal to per-workspace authz at
    this seam.
  </Step>
  <Step>
    **Phase 3 — Rust rewrite (`russh` + `sealant-protocol`).** Reimplement the SSH-server half;
    reuse the protocol crate as the control client (shared types, no codegen drift); ship one static
    musl binary; validate PTY/resize/signals/SFTP/forwarding parity before cutover.
  </Step>
</Steps>

<Callout type="warn">
  **Linchpin risk:** Phase 1's interactive output reliability — if the daemon cannot deliver
  lossless per-session bytes, the whole consolidation stalls.
</Callout>

## Recommendation: do we need a network MITM proxy?

No. The real problem is that capture is **cooperative and bypassable**, not that TLS is opaque.
Completeness is a coverage/enforcement problem, not a decryption one — solve it with a per-workspace
netns plus eBPF/conntrack plus DNS observation (non-intercepting), not a CA-injecting MITM.

### What exists today

Everything that captures network traffic is one thing: a **cooperative, in-process HTTP/HTTPS
forward proxy** (roughly 640 lines, `sealant-network/src/`). A TCP listener on `127.0.0.1:0`;
children are pointed at it purely by `HTTP_PROXY` / `HTTPS_PROXY` env vars — **env injection is the
entire routing mechanism** (no iptables, no netns, no TPROXY, no `SO_ORIGINAL_DST`). Plain HTTP
exposes method/host/path/status/bytes; HTTPS `CONNECT` exposes only host/port/bytes (TLS is piped
verbatim, never inspected). Of the five `NetworkMode`s, `Metadata` / `Privileged` / `Payload` are
**unimplemented** and silently degrade to `Proxy` absent `CAP_NET_ADMIN` — so `Proxy` is the only
mode that ever runs.

### The gaps

| Blind spot                   | Status          | Why                                                                          |
| ---------------------------- | --------------- | ---------------------------------------------------------------------------- |
| TLS/HTTPS payloads and paths | invisible       | CONNECT tunnels copied byte-for-byte; no decryption (by design)              |
| **Any proxy bypass**         | fully invisible | cooperative only — unset the env var and you are unseen (threat-model row 7) |
| UDP / QUIC / HTTP-3          | invisible       | proxy is TCP-only                                                            |
| DNS                          | invisible       | `metadata` mode unimplemented; no resolver hook                              |
| Raw / non-HTTP TCP           | invisible       | only HTTP + CONNECT verbs handled; no transparent redirect                   |
| Per-process attribution      | weak            | rides on the proxy connection's `execution_id`, not kernel socket ownership  |

The repo is already honest about this: the proxy is a **cooperative evidence source, not an enforced
egress boundary** (threat-model, known-limitations, ADR-0009).

<Callout type="idea">
  **Verdict: do not build a TLS MITM as the path to "capture everything."** MITM solves only 1 of 6
  blind spots (payloads) while leaving the dominant gap — bypass — wide open, and it imports the
  heaviest costs (a trusted CA injected into the workspace, decrypted secrets, pinning breakage). It
  is the wrong tool for completeness.
</Callout>

### Recommended capture strategy

In priority order.

<Cards>
  <Card title="A — Capture all metadata (not MITM)">
    Per-workspace netns plus eBPF/conntrack plus DNS. This is the unbuilt `privileged` / `metadata`
    the architecture already anticipates (ADR-0009). eBPF on `connect` / `sendmsg` (or
    conntrack/netlink) captures every TCP and UDP flow with reliable PID attribution — closing
    bypass, UDP/QUIC, raw sockets and attribution at once, without touching payloads or needing a
    CA. Add a DNS observer for queried names.
  </Card>
  <Card title="B — Make capture enforceable">
    Transparent redirect inside the netns. `nftables` REDIRECT/TPROXY forces all egress through the
    local proxy (recover the destination via `SO_ORIGINAL_DST` / TPROXY), converting cooperative
    evidence into an actual chokepoint — still without decrypting TLS.
  </Card>
  <Card title="C — TLS MITM (payload mode), gated">
    Build only if a product genuinely needs request/response bodies, and only opt-in. The right
    shape is netns plus redirect into a terminating proxy with a per-workspace, short-lived CA
    trusted only inside that workspace, a strict host allow/deny list, and mandatory redaction.
    **Off by default** — it weakens the workspace's own TLS trust, breaks pinned clients, and turns
    the collector into a plaintext-secret aggregator.
  </Card>
</Cards>

<Callout type="info">
  **The real critical path** is capability conveyance: A and B are blocked on `CAP_NET_ADMIN` /
  `CAP_BPF` reaching the container plus a separate, isolated `sealant-net-collector` privileged
  process (ADR-0009 already designs this — narrow IPC, no raw packets, keeping the main daemon
  unprivileged). Land that first; it yields near-complete, attributable, non-intercepting metadata
  at far lower trust/perf/complexity cost than MITM.
</Callout>

## How it all fits together

The three recommendations are not independent — they share the daemon as the center of gravity, and
they have a natural order. The unifying theme: **the daemon already implements the hard parts**
(supervision, a session/exec API, capture and redaction) — the work is to _route through it_ rather
than around it.

<Steps>
  <Step>
    **First, persist the telemetry that already exists.** The cheapest, lowest-risk win: the capture
    pipeline is built and dead-ends in memory. Wiring a real consumer (DB and/or rabbitmq) makes the
    daemon valuable on its own and is a prerequisite for the gateway's later observability story. No
    new daemon capability needed.
  </Step>
  <Step>
    **In parallel, fix the gateway's host-key trust now (Phase 0).** Independent of everything else
    and exploitable today — it must not wait for any rewrite.
  </Step>
  <Step>
    **Then clean up image building — because it unblocks the gateway consolidation.** Moving
    orchestration into `sealantd boot` makes the daemon PID 1 and the owner of process/PTY
    lifecycle, which is exactly the precondition for the gateway to tunnel sessions to the daemon
    socket and delete the in-container `sshd`.
  </Step>
  <Step>
    **Next, grow the daemon's session API and consolidate the gateway onto it (Phases 1–2).** Add
    the reliable per-session output stream plus `forwardTcp` plus SFTP, then point the gateway at
    `/run/sealantd.sock`. This removes the second SSH hop and unifies the two transports into one.
  </Step>
  <Step>
    **Then unlock real network coverage (A/B).** Capability conveyance plus the isolated collector
    plus eBPF/conntrack/DNS and a netns redirect — a larger, privilege-sensitive effort that builds
    on the now-persisted telemetry sink from step 1.
  </Step>
  <Step>
    **Last, the Rust rewrites.** Delete the generated bash and rewrite the gateway in `russh` plus
    `sealant-protocol` — genuinely valuable, but only after the designs above are proven, so the
    type-sharing win is real and no flawed shape gets re-encoded in a new language.
  </Step>
</Steps>

<Callout type="info">
  **One sentence:** make the daemon the durable sink and the in-container owner of process plus
  session lifecycle first; route the gateway and the network capture through it second; reach for
  Rust and MITM last, and only where they pay for themselves.
</Callout>
