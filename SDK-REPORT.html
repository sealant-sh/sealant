2 · Architecture & the bottom-up read path

The SDK is a thin, ergonomic facade over the existing platform. Reads of run details flow through four layers you can build and test independently — exactly the order requested:

READ PATH (run details / execution record) SDK facade @sealant/sdk run.record.replay() / timeline() / scrollback() / loss() │ HTTP (contract-derived client) ▼ API layer apps/api GET /v1/runs/:id, /timeline, /scrollback, /loss │ ▼ Contract @sealant/api-contracts RunsGroup (Effect HttpApi + Schema) │ ▼ Query repo @sealant/db RunRepo + RunTelemetry (folds the telemetry log) │ ▼ Postgres runs · telemetry_events · telemetry_timeline · telemetry_scrollback · …

The write/exec path (create + run) reuses the platform spine and the proven docker-exec transport:

WRITE / EXEC PATH sandboxes.create({repository, harness}) ──HTTP──▶ POST /v1/sandboxes ──▶ worker builds image + launches container sandbox.harness.run(prompt) ──┬─ POST /v1/runs (register the run) ├─ docker exec ─▶ sealantd ─▶ opencode run "<prompt>" ├─ fork TelemetryIngester (2nd connection) ─▶ persist record └─ capture git diff ─▶ PATCH /v1/runs/:id (completed)

The SDK’s three strata

1 · Promise facade

Plain async/await objects (Sealant, Sandbox, Run, RunRecord). The public surface. No Effect leaks: tagged failures are mapped to plain Errors.

2 · Effect core

The app Layer (wire client + sealantd runtime + telemetry ingester) built once into a memoized Scope; operations run against it and surface plain errors.

3 · Wire (derived, not written)

HttpApiClient.make(ControlPlaneAPI) — a fully-typed client generated from the @sealant/api-contracts contract. No codegen artifact, no drift; it replaces the hand-rolled fetch+zod client.

Harness-neutral

opencode(), codex(), claudeCode(), customHarness() — thin values describing how to invoke a harness one-shot.

3 · What was built, layer by layer

① Query repository — @sealant/db

↳
packages/db/src/repositories/runs.ts — RunRepo: create / get / list runs + markRunning/markCompleted/markFailed, on a typed TaggedError channel (mirrors SandboxAttemptRepo). The execution record (RunTelemetry) already folds the append-only telemetry log.
Why: a run is one harness execution and needs first-class storage so its record survives the sandbox and is queryable, not parsed.

② Contract — @sealant/api-contracts

↳
packages/api-contracts/src/core-api/runs.ts — the RunsGroup Effect HttpApi group: POST/GET/PATCH /v1/runs and GET /v1/runs/:id/{timeline,scrollback,loss}, with Schema request/response types and tagged errors. Added to ControlPlaneAPI.
Why: the single source of truth. The SDK client is derived from this; uint64 sequences travel as decimal strings to survive JSON.

③ API layer — apps/api

↳
apps/api/src/routes/runs/ — handlers backed by RunRepo (metadata) and TelemetryQuery (timeline / byte-exact scrollback / provenance-honest loss report), wired into the server with a self-contained telemetry layer.
Why: makes the record a network resource, so the SDK is a real client and the record outlives the sandbox.

④ SDK — @sealant/sdk (new package, ~1,660 LoC)

↳
The full fluent surface (src/types.ts), the derived wire client + runtime (src/effect/), the facade objects (src/facade/), create()’s blueprint lowering (src/internal/blueprint.ts), and the host-local harness.run() orchestration (src/effect/run-harness.ts): resolve the container → register the run → fork the ingester → exec the harness → capture the diff → close the record → return a queryable Run.
Why: this is the public product surface — the thing developers import and build on.

Data-model change: the first-class runs table

A new runs table (one row per harness execution) was added, and all six telemetry foreign keys were re-pointed from sandbox_attempts.id to runs.id (migration 20260627205522_runs_table_and_telemetry_fk, applied + verified). Telemetry ingestion is now run-initiator-owned: the SDK forks the ingester per run, so the worker’s old per-sandbox auto-ingest was retired to a Phase-3 rework.

5 · The two decisions & the model correction

DecisionChoiceWhy it mattered
Slice-1 scope Host-local thin slice Fastest path to a real demo: SDK on the Docker host, telemetry ingested in-process, no auth. Network/auth deferred behind unchanged public types.
Run identity First-class runs entity, now A harness run is distinct from a sandbox provisioning attempt; modelling it properly up front keeps runs.get(id) correct and lets one sandbox host many runs.
A correction worth surfacing: the plan framed runId == attempt as a “hacky conflation.” Reading the code showed it’s actually a pervasive, intentional convention (runtime instances, build jobs, run-links all key on the attempt id). So re-pointing only telemetry to a new table was done deliberately — telemetry now belongs to a harness run; everything else keeps the existing convention. No split-brain.

6 · The end-to-end proof

The verbatim public API — unchanged from the marketing copy:

import { Sealant, opencode } from "@sealant/sdk";

const sealant  = new Sealant({ baseUrl: "http://localhost:4000" });
const sandbox  = await sealant.sandboxes.create({ repository, harness: opencode() });
const run      = await sandbox.harness.run("Append a line … to the README file.");
await run.record.replay();                         // scrubable timeline entries
console.log(await run.record.transcript());        // …or the readable replay (commands, not noise)

Run for real against the live stack (Postgres + RabbitMQ + zot + control-plane API + worker), it produced:

8 · Honest status & what’s deferred

The SDK is feature-complete for the host-local slice. In the spirit of the product (“it reports; it does not judge — and it’s honest about provenance”), here is exactly what is and isn’t real yet:

Real & validated
  • sandboxes.create / get / list
  • harness.run() one-shot (real agent)
  • run.record.replay / timeline / scrollback / loss / summary
  • runs.get() — read a past run
  • derived, type-safe wire client
  • full run-detail read path through all 4 layers
Typed now, deferred by phase
  • record.stream() live tail — Phase 2 (delivery integrity)
  • harness.start() / session() — Phase 2/3
  • stop / restart / expire, sandbox.events() — Phase 1/3
  • fileTreeAt / processTreeAt folds — Phase 1
  • publishable dist + /effect subpath — Phase 4

All reject with a clear SealantNotImplementedError, so callers compile against the final shape today.

One honest provenance note. In the demo loss.complete reads false. This is not dropped data — the 16 entries are all captured. The host-local ingester connects to the daemon mid-life and the run’s epoch closes via stream-end rather than an observed clean daemon STOPPED, so the record conservatively flags that it couldn’t prove clean closure. True clean-close + ack/resume is precisely the Phase-2 “delivery integrity” work the plan scopes. The record reporting its own uncertainty is the product behaving correctly.

Slice-1 is host-local and unauthenticated by design (the SDK runs on the Docker host with DB access for in-process ingestion). The public constructor is already { baseUrl, apiKey }; host-local needs (ownerUserId, registry, databaseUrl) live in an internal config and never leak into the published type, so adding auth + remote reads later changes nothing for callers.

docker compose up -d

# 2 · apply migrations
DATABASE_URL=postgresql://sealant:sealant@127.0.0.1:5433/sealant_control_plane \
  pnpm --filter @sealant/db db:migrate

# 3 · control-plane API (host, :4000)
pnpm --filter @sealant/api dev

# 4 · build/launch worker (host; needs docker + buildx)
DEFAULT_SSH_AUTHORIZED_KEYS_FILE=$PWD/.secrets/authorized_keys \
  pnpm --filter @sealant/worker dev
