Design brief

Reusing a developer's gh / claude / codex logins inside sandboxes

How to let people use their existing tools — including paid Claude & Codex subscriptions — inside a sandbox without setting them up every time. Acquire, store, inject.

Date 2026‑06‑29 Branch feat/self-host-installer Repos Sealant · get‑sealant/sealantd gh capture token (no GitHub App) Status design · not built

§Executive summary

Three independent questions — where tokens come from, where we store them, how we pass them to containers — map to three independent layers. Keeping them decoupled lets each evolve on its own.

  • The first SDK MVP (token → docker run -e) is dead. The in‑sandbox daemon deliberately strips secret‑named env vars from the agent. Confirmed in code.
  • The real injection seam already exists in sealantd: per‑exec ExecArgs.env (harness‑scoped, unfiltered) for tokens + the existing open_sftp channel for credential files. Near‑zero daemon change.
  • The store is the durable seam: an encrypted Postgres credential store (AES‑256‑GCM envelope, no new deps). The create request carries a reference, never the secret — fixing the MVP's plaintext‑in‑spec leak.
  • Acquisition is resolved per provider: gh = capture the user's token; claude = claude setup-token (one browser flow, ~1yr); codex = API key for workers, ChatGPT login only as a single‑owner lease.
  • One honest ceiling: the harness runs as root in the daemon's namespace, so in‑box scoping is blast‑radius reduction, not confidentiality from the agent. Real isolation needs the harness dropped to an unprivileged uid.

§The three layers

ACQUIRE

Capture the login that already exists (disk / vendor CLI), or a one‑time sign‑in.

◆ Trust boundary
ACQUIRE and the master key stay outside the container wall. The daemon inside only ever receives an already‑decrypted, sandbox‑scoped credential — it never holds the store key or reaches the dev's disk.

1The pivotal correction

The sandbox daemon (sealantd, PID 1) composes an exec'd process's environment in layers (sealant-process/src/runtime.rs:169‑182):

command.env_clear();                     // start from empty
for var in &self.config.child_env { … }     // curated harness base env — SECRET-STRIPPED
for var in &args.env { … }              // ExecArgs.env — applied ON TOP, UNFILTERED
for (k,v) in extra_env { … }          // network proxy env

And child_env is the secret‑stripped passthrough — is_secret_key() drops anything containing TOKEN/SECRET/PASSWORD/CREDENTIAL/APIKEY or ending _KEY (boot/config.rs:65‑79, 626‑637).

✕ Why the first MVP is dead
A token shipped via docker run -e (GH_TOKEN, ANTHROPIC_API_KEY, OPENAI_API_KEY, CLAUDE_CODE_OAUTH_TOKEN) lands in PID‑1's environment → gets stripped by the passthrough filter → never reaches the harness. The daemon is built to keep secrets out of the agent's environment. Container‑wide env is the wrong channel, by design.
✓ The correct seam (already in the protocol)
Because Sealant runs foreground = sleep infinity and execs the harness on demand via the control socket, credentials belong in that exec's ExecArgs.env — applied after env_clear + child_env, unfiltered, to only the harness process (not container‑wide, not in docker inspect, not visible to other sandbox processes). File‑style creds use the existing open_sftp channel (sealant.proto:336, 387). No proto change strictly required.
▲ The ceiling that bounds everything
The harness runs as root in the daemon's namespace (child_uid = None, boot/mod.rs:194). A hostile root harness can read /proc/<pid>/environ and our 0600 files regardless. In‑box scoping is blast‑radius reduction, not confidentiality from the agent. Genuine isolation — and any in‑box token refresh — requires first dropping the harness to an unprivileged uid.

One small daemon change is near‑certain regardless: seed the output Redactor (runtime.rs:31‑50, 136‑137) with the injected token values, since ExecArgs.env secrets are not auto‑scrubbed from captured I/O — a harness that echoes a token would otherwise leak it into the event log.

2Acquire — where the tokens come from

A long‑lived credential is already on disk for all three tools, readable with no browser flow — but each has a catch. The optimal source per provider:

ProviderChosen sourceInject asWhy / catch
gh capture Capture the user's existing token: gh auth token → gho_/ghp_ (interactive fallback gh auth login). GH_TOKEN Per your call, no GitHub App. Simpler — no App to operate, no install flow. Trade‑off accepted: broader scope & no auto‑rotation vs. per‑run App tokens. Optionally run gh auth setup-git in‑box for git‑over‑HTTPS.
claude durable claude setup-token → ~1yr token (one browser flow, ever). CLAUDE_CODE_OAUTH_TOKEN Reusing the on‑disk refresh token is out: a Feb 2026 ToS change restricts consumer OAuth tokens to Claude Code / Claude.ai, server‑enforced. setup-token is the only sanctioned long‑lived path.
codex lease OPENAI_API_KEY for any multi‑worker use; ChatGPT login only as a single‑owner lease. OPENAI_API_KEY / ~/.codex/auth.json The ChatGPT refresh token is single‑use / rotating — the moment a second consumer refreshes, the first dies. A stored sub credential must have exactly one live owner; API key is the fan‑out‑safe path.
◆ The honest answer to "without an extra flow?"
gh: yes — capture the on‑disk token, no flow. codex: yes to acquire (read auth.json), but rotation makes it a single‑owner lease. claude: no — the durable, compliant token needs the one‑time setup-token browser flow; the on‑disk‑refresh shortcut is now ToS‑restricted and revocable.

3Store — where we keep them

A real encrypted store in the control plane (Postgres is already in the stack; two dormant secrets tables already exist at packages/db/.../control-plane.ts:399‑444 to sit beside).

Schema — purpose‑built for live, refreshable connections

New principal_credentials (queryable metadata) + principal_credential_versions (ciphertext only). Metadata is safe to return from the API; ciphertext never leaves the DB.

Returned by the API (metadata)Never leaves the DB (ciphertext)
id, provider, kind, label, status, scopes, accountIdentifier, last4, expiresAt, connectedAt, lastRefreshedAt, rotationCount envelope — AES‑256‑GCM(ciphertext + nonce + tag + wrapped DEK); the decrypted access/refresh tokens

Encryption — AES‑256‑GCM envelope, zero new deps

// node:crypto only. KEK → per-version DEK → value. No libsodium, no native build.
write(plaintext, aad = `${ownerUserId}:${credentialId}`):
  dek   = randomBytes(32)                              // fresh per version
  valCt = AES-256-GCM(dek, iv12, plaintext, aad)        // aad binds ciphertext to its row
  wDek  = AES-256-GCM(KEK, iv12, dek)                   // KEK from SEALANT_SECRETS_KEY (→ KMS later)
  envelope = base64(JSON{ v:1, kekId, valIv, valTag, valCt, dekIv, dekTag, wDek })

A KeyProvider interface keeps the master key pluggable: EnvKeyProvider (SEALANT_SECRETS_KEY) for self‑host now, KmsKeyProvider for hosted later. Key rotation = re‑wrap the DEK, never re‑encrypt values.

API & resolver

  • Write‑only API — POST /credentials (connect), GET (metadata only), POST /credentials/:id/revoke. No handler ever returns ciphertext. Scoped to the authed owner.
  • Worker resolver mirrors the existing github-installation-auth-resolver.ts: resolve a sealant-credential:<id> reference, re‑scope to the owner, decrypt, inject. The worker is the single refresher (advisory‑locked) — which dissolves the codex rotation race.
✓ Fixes the MVP leak
The create request carries a reference (credentialSelections → spec.runtime.credentialRefs), never the secret — mirroring how sourceSelection/dotfilesSelection already work. No secret bytes are ever persisted in the spec or the attempt snapshot.

4Inject — into the running container

The worker decrypts at launch and hands the credential to the daemon, which scopes it to the harness:

  • Tokens → ExecArgs.env on the harness exec (harness‑scoped, unfiltered).
  • Files → open_sftp — written 0600 under /root/.claude, /root/.codex before the harness exec.

◇ Refresh ownership
Slice 1: sealantd does not hold refresh tokens — the worker refreshes and re‑provisions. Env tokens are pinned per harness‑exec (env binds at spawn); only file‑based creds are refreshable mid‑session. In‑box refresh is deferred until the uid drop.

5The sealant CLI / TUI

A new apps/cli package — opentui (React bindings) on the Bun runtime (its core loads via FFI/dlopen; Node would need --experimental-ffi). Every command keeps a non‑TTY path (--json) so the core never requires a terminal.

# acquire once → store → reference later, anywhere (incl. the worker)
sealant login claude        # spawn `claude setup-token`, capture sk-ant-oat01…, encrypt → handle
sealant ps                   # list sandboxes (slice 0 — read-only over the existing SDK)

await sealant.sandboxes.create({
  repository: "github.com/acme/billing-service",
  harness: claudeCode(),
  use: ["claude:max", "gh:default"],   // stored handles, resolved worker-side at launch
})

Keep both paths: forward (capture my live host login now, ephemeral) and use (resolve a stored handle, durable/portable). They share the same downstream injection.

7Decisions & forks

Worker‑side vs. CLI‑side resolution Worker‑side + server‑side Postgres store — required for codex leases & the async model. recommend · open

8Staged build order

  1. 0sealant ps, read‑only over the existing SDK. Ships the binary + opentui/Bun packaging story before any credential complexity.
  2. 1Encrypted store + sealant login gh (capture). Validates encrypt/decrypt, the key model, and the handle.
  3. 2create({ use }) end‑to‑end with gh — reference → worker resolve → ExecArgs.env injection. Plus the Redactor seed in sealantd.
  4. 3sealant login claude via setup-token — the first interactive browser‑wait acquire; file/sftp injection.
  5. 4sealant login codex — API key first, then the ChatGPT single‑owner lease with the rotation guard.
  6. 5Isolation track — drop the harness to an unprivileged uid; unlocks in‑box refresh + real confidentiality.
