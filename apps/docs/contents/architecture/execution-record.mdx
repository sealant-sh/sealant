Every run inside a Sealant sandbox produces an execution record: a durable, faithful account of
everything that happened. `sealantd` runs as PID 1 inside the sandbox and emits a typed telemetry
firehose; the `@sealant/telemetry` package consumes that firehose and persists every event as an
append-only, event-sourced log on the control-plane Postgres database, behind a pluggable storage
seam.

<Callout type="info">
  The log is the system of record. Read models — terminal scrollback, a run timeline, loss
  accounting — are *derived* from it and are safely disposable. A new view (for example "file tree
  at step N") is a re-fold of the existing log, never a re-capture. Lossless, ordered, and
  replayable capture is the precondition for everything else.
</Callout>

  </Card>
  <Card title="Why this shape">

- New package `@sealant/telemetry`, built with Effect (services-from-context, layers).
- Six new Postgres tables in `@sealant/db`: one log, one epoch, one blob store, and three
  projection/loss tables.
- A dedicated telemetry worker-kind in `apps/worker` that runs one ingester per running sandbox.
- Storage defaults to Postgres; the `TelemetrySink` seam lets ClickHouse or Redis slot in later with
  no ingester change.
- A real-Postgres integration test proves the whole slice end to end, alongside the pure unit suite.

## Goal & north star

Sealant's promise is reproducible sandbox runs and issue-to-PR workflows with clear reporting of
what happened. That reporting needs raw material: a durable, faithful record of everything that
occurred inside a sandbox. The execution record is not just a log file — its data model is dictated
by three long-term goals.

<Steps>
  <Step>
    **Time-travel** — step backward and forward through an agent's execution and reconstruct exact
    state at any point: terminal scrollback byte-for-byte, file tree, process tree, and in-flight
    network.
  </Step>
  <Step>
    **A proper PR timeline** — what the agent actually did: commands, edits and diffs, network
    egress, exit reasons, and durations.
  </Step>
  <Step>
    **Arbitrary granularity** — from a single byte up to whole-run rollups, served from the same
    record.
  </Step>
</Steps>

<Callout type="info">
  **Consequence:** lossless, ordered, replayable capture is non-negotiable. An append-only event log
  keyed on `(runtimeId, sequence)`, with `eventId` dedup and the raw envelope retained, is the only
  shape that does not foreclose the goals above. The current implementation need not *implement*
  time-travel — but it must *store* everything required to derive it later.
</Callout>

## The sealantd telemetry surface

`sealantd` emits a typed Protobuf telemetry stream. The unit is `sealant.v1.EventEnvelope`: a set of
correlation and ordering fields followed by a discriminated payload (a `oneof` with twelve cases).
The wire contract is published as `@sealant/runtime-protocol`.

### Envelope metadata — correlation & ordering keys

| Field                                                   | Type      | Role                                                                           |
| ------------------------------------------------------- | --------- | ------------------------------------------------------------------------------ |
| `eventId`                                               | `string`  | Globally-unique id — the absolute dedup key, used as the row primary key.      |
| `runtimeId`                                             | `string`  | The daemon instance. `sequence` is monotonic _per_ `runtimeId`.                |
| `executionId` / `sessionId` / `processId` / `requestId` | `string?` | Optional finer correlation: exec scope, PTY session, process, causing request. |
| `sequence`                                              | `uint64`  | Per-runtime monotonic counter — the canonical ordering and gap-detection key.  |
| `observedAt`                                            | `int64`   | Wall-clock (skewable) — for display only.                                      |
| `monotonicTimestamp`                                    | `uint64`  | Skew-proof ordering tiebreaker.                                                |
| `schemaVersion`                                         | `uint32`  | Per-envelope schema version, for drift tolerance.                              |
| `captureMethod` / `confidence`                          | `enum`    | Provenance: how it was observed (pipe / pty / ebpf / …) and how trustworthy.   |

### The twelve payload cases

| Case                    | Carries                                                                                           | Volume                    |
| ----------------------- | ------------------------------------------------------------------------------------------------- | ------------------------- |
| `processStarted`        | pid, pgid, executable, args, cwd, startedAt                                                       | 1 per process             |
| `processExited`         | exitCode?, signal?, reason, durationMicros                                                        | 1 per process             |
| `ioChunk`               | stream, byteCount, `streamOffset`, content?, artifact?, `transform{redacted,truncated,coalesced}` | **very high** (the bytes) |
| `fileChange`            | kind, path, renameFrom?, entry, certain                                                           | high (bursty)             |
| `fileDiffAvailable`     | added, modified, deleted, renamed                                                                 | periodic                  |
| `fileSnapshotCompleted` | root, fileCount                                                                                   | occasional                |
| `fileWatchOverflow`     | root (watch queue overflowed, state stale)                                                        | rare (loss signal)        |
| `networkRequest`        | scheme, method?, host, port, path?, status?, bytesSent, bytesReceived, durationMicros             | per request               |
| `networkSourceObserved` | host, resolvedIps[], port, scheme?                                                                | per connection            |
| `runtimeStateChanged`   | state (STARTING…STOPPED), reason?                                                                 | lifecycle                 |
| `runtimeHeartbeat`      | state                                                                                             | periodic                  |
| `telemetryDropped`      | reason, **count**, priority                                                                       | rare (loss signal)        |

There is also a pollable control plane that is not part of the event stream: `HealthReport` (queue
depth, spool bytes, dropped/redacted/coalesced counters, `sinkConnected`), `RuntimeMetrics`, and
`Capabilities`. `ioChunk` is the one unbounded, content-bearing event — its bytes are what get
offloaded to the artifact store.

## How the SDKs are used

There are three layers between the daemon and the ingester, and the highest one is the tap point.

| Package                     | Provides                                                                                                                                                            | Used for                                                |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------- |
| `@sealant/runtime-protocol` | Typed Protobuf codec and framing (`EventEnvelope`, all message and enum types, schemas)                                                                             | Types plus `create()` for test fixtures                 |
| `@sealant/runtime-client`   | `SealantClient`: a transport-agnostic control client. `client.events()` is an async iterator over decoded envelopes; `fromStream()` drives it over any Node Duplex. | The event firehose; `StreamKind` / `RuntimeState` enums |
| `@sealant/sandboxes`        | `SealantRuntime` / `SealantSession` — an Effect wrapper that adapts `client.events()` into an Effect `Stream` and owns connection lifecycle via `Scope`.            | **The tap point**                                       |

### The tap point

`packages/sandboxes/src/sealantd/runtime.ts` wraps the SDK iterator into an Effect stream — this is
what the ingester subscribes to. It never touches raw Protobuf or the socket.

```ts
// packages/sandboxes/src/sealantd/runtime.ts — the SDK iterator becomes an Effect Stream
const makeSession = (client: SealantClient): SealantSession => ({
  health: withSealantError(
    "health",
    Effect.promise(() => client.health()),
  ),
  // Stream.fromAsyncIterable pulls one event per next() — the SDK iterator is the backpressure boundary.
  events: Stream.fromAsyncIterable(client.events(), (c) => mapSealantError("events", c)),
  // …exec / writeStdin / shutdown …
});

// connect() acquires the transport + client under a Scope; the finalizer closes them in reverse.
connect: (target) =>
  Effect.gen(function* () {
    const duplex = yield* transport.open(target); // docker exec -i + socat → unix socket
    const client = yield* Effect.acquireRelease(
      Effect.sync(() => SealantClient.fromStream(duplex)),
      (c) => Effect.sync(() => c.close()),
    );
    return makeSession(client);
  });
```

### What the ingester does with it

The ingester opens its own, second control connection. It never shares `execInSandbox`'s
single-consumer iterator (which would steal events) and never touches the gateway's byte channels.
It then drains the whole runtime stream for the connection's lifetime as a single `Effect.gen`
program.

```ts
// packages/telemetry/src/ingester.ts — the entire consume → persist pipeline
const session = yield* runtime.connect(target);       // SealantSession (Scoped)
const runtimeId = (yield* session.health).runtimeId;  // authoritative id from the SDK
yield* sink.openEpoch({ runId, runtimeId, schemaVersion: 0 });

yield* session.events.pipe(
  Stream.tap((env) => /* detect sequence gaps + persist telemetryDropped / overflow loss spans */),
  Stream.groupedWithin(256, "250 millis"),            // bound the SDK's otherwise-unbounded queue
  Stream.mapEffect((batch) =>
    sink.appendBatch({ runId, runtimeId, batch: batch.map(normalizeEnvelope) })
      .pipe(Effect.retry(Schedule.exponential("100 millis").pipe(Schedule.both(Schedule.recurs(5))))),
  ),
  Stream.runDrain,
).pipe(
  Effect.ensuring(/* close epoch; stream-end is SUSPICIOUS → record early_close unless a clean STOPPED was seen */),
);
```

<Accordions>
  <Accordion title="Why a dedicated second connection instead of fanning out one stream?">
    The SDK's `events()` is a single-consumer iterator, and `Stream.broadcastDynamic` is not
    available in this Effect build — so an in-SDK fan-out would be more work and riskier than simply
    opening a second control channel. Opening a dedicated connection keeps the blast radius on the
    gateway and build worker at zero.
  </Accordion>
</Accordions>

## Architecture

End to end, the log is the system of record and everything else is derived from it.

```text
  sealantd (PID 1, in sandbox)
        │  events()  ── typed EventEnvelope firehose
        ▼
  docker exec -i + socat ──► SealantRuntime.connect(target)        [OWN 2nd control connection]
        │                         SealantSession.events : Stream<EventEnvelope>
        ▼
  ┌─────────────────────────── @sealant/telemetry ───────────────────────────┐
  │  TelemetryIngester      gap-detect → groupedWithin(256, 250ms) → append    │
  │        │                                                                   │
  │        ▼   normalizeEnvelope()  (pure: split bytes, stringify bigints)     │
  │  TelemetrySink  (pluggable Tag — Postgres reference adapter)               │
  │        │   appendBatch = ONE tx:                                           │
  │        │     • ioChunk bytes → ArtifactStore.put (sha256, content-addr.)   │
  │        │     • INSERT telemetry_events … ON CONFLICT (runtime_id,sequence) │
  │        │              DO NOTHING RETURNING                                 │
  │        │     • project committed rows → timeline + scrollback (same tx)    │
  │        ▼                                                                   │
  │  Postgres:  telemetry_events (LOG, source of truth)                        │
  │             + run_epochs + artifacts + scrollback + timeline + loss_spans  │
  │        │                                                                   │
  │        ▼   read side                                                       │
  │  TelemetryProjector.rebuild()   ·   TelemetryQuery   ·   RunTelemetry SDK  │
  └───────────────────────────────────────────────────────────────────────────┘
```

The three roles in the diagram:

- **Append-only log** (`telemetry_events`) — immutable, the source of truth.
- **Projections** (`scrollback`, `timeline`) — disposable and rebuildable from the log.
- **Loss accounting** (`loss_spans`) — first-class record of every known gap.

## Design decisions

<Accordions>
  <Accordion title="The event-sourced log is the source of truth">
    The `EventEnvelope` already carries `eventId`, `sequence`, and `monotonicTimestamp` — it *is* an
    event log. It is stored append-only and everything else is derived from it, so a new granularity
    is a re-fold, not a re-capture. This was the decisive property in the design study.
  </Accordion>
  <Accordion title="Pluggable storage seam (TelemetrySink)">
    Rather than commit to a single backend, the engine sits behind a `TelemetrySink` Tag, modeled on
    the repo's existing `SealantTransport` seam. The current build ships a `PostgresTelemetrySink`
    (zero new infrastructure, insert-time idempotent dedup, immediate read-after-write). A
    `ClickHouseTelemetrySink` or `RedisTelemetrySink` can be added later without touching the
    ingester, projector, or query layers.
  </Accordion>
  <Accordion title="Idempotent at-least-once delivery (insert-time dedup)">
    `INSERT … ON CONFLICT (runtime_id, sequence) DO NOTHING RETURNING` means a replayed batch never
    double-writes, and only the newly-committed rows get projected, so projection application is
    idempotent too. Loss spans use deterministic, content-derived ids for the same reason. A bug
    here was caught by the integration test's re-ingest assertion and fixed.
  </Accordion>
  <Accordion title="Projections are rebuilds, not re-captures">
    The ingest path and the rebuild path call the *same* pure derivation functions
    (`deriveTimelineRow`, `deriveScrollbackRow`) over the same stored shape. So
    `TelemetryProjector.rebuild()` reproduces projections byte-identically — a tested invariant.
    This is what makes read models safely disposable.
  </Accordion>
  <Accordion title="bigint discipline and content offload">
    Every `uint64` / `int64` is stored as a Postgres `bigint` (drizzle `mode: "bigint"`), never a JS
    `number`, which would silently narrow past `2^53`. The jsonb payload stringifies bigints.
    `ioChunk` content — the unbounded thing — is offloaded to a content-addressed artifact store, so
    the log row stays bounded while the bytes are never lost.
  </Accordion>
  <Accordion title="Loss is first-class">
    Three signals feed `telemetry_loss_spans`: explicit `telemetryDropped` markers, inferred
    `sequence` gaps (per `runtimeId`), and `fileWatchOverflow`. Stream-end is treated as
    *suspicious* — an `early_close` span is recorded unless a clean terminal `STOPPED` was observed.
    A daemon restart means a new `runtimeId`, which means a new epoch row, never a false gap.
  </Accordion>
</Accordions>

## Time-travel design

`(runId, sequence)` _is_ the time-travel coordinate. Because the full raw envelope is retained
(typed bigint metadata, jsonb payload, `streamOffset`, and transform flags) and content bytes are
kept in the artifact store, any state at any coordinate is a pure fold of the log up to that
`sequence`.

<Cards>
  <Card title="Terminal scrollback (byte-exact) — shipped">
    Select scrollback rows for a `(processId, stream)` where `sequence ≤ atSequence`, order by
    `streamOffset` (not arrival), and resolve each chunk's bytes from the artifact store. Coalesced
    or dropped spans show as explicit gaps, never corrupted concatenation.
  </Card>
  <Card title="File tree at a point — future fold">
    Fold `fileChange` and `fileSnapshotCompleted` up to `atSequence`.
  </Card>
  <Card title="Process tree at a point — future fold">
    Fold `processStarted` and `processExited` by `processId`.
  </Card>
  <Card title="In-flight network — future fold">
    Requests with no completion at or before `atSequence`.
  </Card>
</Cards>

<Callout type="idea">
  **What is stored now so none of this is foreclosed:** the full envelope and payload for every
  event (including unmodeled cases as `payload_case = 'unknown'`); `streamOffset`, `byteCount`, and
  transform flags even when content is absent; the actual `ioChunk` bytes (a no-op offloader was
  explicitly rejected); bigint columns; append-only storage with insert-time dedup; and `runtimeId`
  / `executionId` persisted for the first time, closing the daemon-to-product-run correlation gap.
</Callout>

## Database schema

Six tables were added to `@sealant/db` (drizzle, snake_case). All carry
`FK run_id → sandbox_attempts(id) ON DELETE CASCADE`.

### telemetry_events — log, source of truth

One append-only row per `EventEnvelope`. Never updated or deleted (except FK cascade).

| Column                                                      | Type          | Note                                                           |
| ----------------------------------------------------------- | ------------- | -------------------------------------------------------------- |
| `event_id`                                                  | `text PK`     | `envelope.eventId` — absolute global dedup                     |
| `run_id`                                                    | `text FK`     | references `sandbox_attempts.id` (the product run)             |
| `runtime_id`                                                | `text`        | daemon instance (persisted for the first time)                 |
| `execution_id` / `session_id` / `process_id` / `request_id` | `text?`       | finer correlation                                              |
| `sequence`                                                  | `bigint`      | per-runtime monotonic — ordering and dedup                     |
| `observed_at` / `monotonic_timestamp`                       | `bigint`      | wall-clock / skew-proof tiebreaker                             |
| `schema_version`, `capture_method`, `confidence`            | `integer`     | version and provenance (`0` = UNSPECIFIED preserved)           |
| `payload_case`                                              | `text`        | `'ioChunk'` … `'unknown'`                                      |
| `payload`                                                   | `jsonb`       | full discriminated value (bigints stringified, bytes stripped) |
| `ingested_at`                                               | `timestamptz` | when it was stored                                             |

**Indexes:** `UNIQUE (runtime_id, sequence)` powers the `ON CONFLICT` dedup; plus
`(run_id, sequence)`, `(run_id, process_id, sequence)`, and `(run_id, payload_case, sequence)`.

### telemetry_run_epochs — log

One row per (run, daemon instance). Closes the `runtimeId`-to-`runId` gap and holds watermarks plus
a close reason. A daemon restart opens a new epoch (a hard discontinuity, never a gap).

| Column                             | Type          | Note                                    |
| ---------------------------------- | ------------- | --------------------------------------- |
| `id`                               | `text PK`     | `tep_<runId>_<rtShort>`                 |
| `run_id` / `runtime_id`            | `text`        | `UNIQUE (run_id, runtime_id)`           |
| `status`                           | `enum`        | open / closed / lost                    |
| `first_sequence` / `last_sequence` | `bigint?`     | watermark                               |
| `events_persisted`                 | `bigint`      | counter                                 |
| `close_reason`                     | `enum?`       | stream-end / transport-close / shutdown |
| `opened_at` / `closed_at`          | `timestamptz` | lifecycle                               |

### telemetry_artifacts — content store

Content-addressed blob store for `ioChunk` bytes (and daemon `ArtifactRef` bodies). Deduped by
content; the current build stores bytes inline.

| Column                         | Type               | Note                                                           |
| ------------------------------ | ------------------ | -------------------------------------------------------------- |
| `id`                           | `text PK`          | `tart_<runId>_<algo>_<hash>`                                   |
| `run_id`                       | `text FK`          | cascade                                                        |
| `algo` / `hash`                | `text`             | sha256 (ours) or `ArtifactRef` — `UNIQUE (run_id, algo, hash)` |
| `byte_size`                    | `bigint`           | equals `IoChunk.byteCount`                                     |
| `storage_backend`              | `enum`             | inline (current) / s3 / filesystem / … (reuses existing enum)  |
| `storage_key` / `inline_bytes` | `text?` / `bytea?` | key for external backends; inline body for the current build   |

### telemetry_scrollback — projection, rebuildable

Byte-exact terminal reassembly key. Reconstructed by ordering on `streamOffset` and resolving
content from the artifact store.

| Column                                 | Type      | Note                                                 |
| -------------------------------------- | --------- | ---------------------------------------------------- |
| `event_id`                             | `text PK` | one row per `ioChunk`                                |
| `run_id` / `process_id` / `session_id` | `text`    | scope                                                |
| `stream`                               | `integer` | `StreamKind` (stdout / stderr / pty …)               |
| `stream_offset` / `byte_count`         | `bigint`  | `UNIQUE (run_id, process_id, stream, stream_offset)` |
| `content_algo` / `content_hash`        | `text?`   | natural key into `telemetry_artifacts`               |
| `redacted` / `truncated` / `coalesced` | `boolean` | fidelity flags                                       |
| `original_byte_count`                  | `bigint?` | true pre-transform size                              |
| `sequence`                             | `bigint`  | for the time-travel ceiling                          |

### telemetry_timeline — projection, rebuildable

Per-run "what happened": one ordered, human-summarized entry per event. The PR-report spine.

| Column                | Type              | Note                                           |
| --------------------- | ----------------- | ---------------------------------------------- |
| `event_id`            | `text PK`         |                                                |
| `run_id` / `sequence` | `text` / `bigint` | `UNIQUE (run_id, sequence)`                    |
| `kind`                | `text`            | equals `payload_case`                          |
| `occurred_at`         | `bigint`          | `monotonicTimestamp`                           |
| `summary`             | `text`            | e.g. `exec /bin/sh -c echo hi`, `stdout 3B @0` |
| `ref_json`            | `jsonb?`          | compact payload reference                      |

### telemetry_loss_spans — loss accounting

A first-class record of every known gap, so completeness is auditable rather than silently assumed.

| Column                                            | Type                 | Note                                                        |
| ------------------------------------------------- | -------------------- | ----------------------------------------------------------- |
| `id`                                              | `text PK`            | deterministic (idempotent across re-ingest)                 |
| `run_id` / `runtime_id`                           | `text`               | scope                                                       |
| `kind`                                            | `enum`               | dropped_event / sequence_gap / watch_overflow / early_close |
| `from_sequence` / `to_sequence` / `dropped_count` | `bigint?`            | span extent                                                 |
| `priority` / `reason`                             | `integer?` / `text?` | detail                                                      |
| `detected_via`                                    | `enum`               | marker (daemon told us) / gap (inferred)                    |

## The consumer SDK surface

A single Effect service, `RunTelemetry`, is the facade a future product SDK imports. The current
read methods are implemented; the time-travel and rollup methods are typed now but fail with a
`NotImplemented` error — so the surface is stable and callers compile, but nothing is half-built.

```ts
interface RunTelemetryService {
  // ── Implemented today ──────────────────────────────────────
  listRuns: (f?) => Effect<readonly RunSummary[]>;
  getTimeline: (runId, opts?) => Stream<TimelineEntry>;
  getEvent: (runId, sequence) => Effect<TelemetryEvent>;
  getLossReport: (runId) => Effect<LossReport>;
  reconstructScrollback: (runId, processId, stream, atSequence) => Stream<Uint8Array>;

  // ── Future (typed; each is a later fold of the same log) ────
  getRunRollup: (runId) => Effect<RunRollup>;
  reconstructFileTree: (runId, atSequence) => Effect<FileTreeSnapshot>;
  reconstructProcessTree: (runId, atSequence) => Effect<ProcessTree>;
  reconstructNetworkInFlight: (runId, atSequence) => Effect<NetConn[]>;
  tail: (runId, fromSequence?) => Stream<TimelineEntry>;
}
```

Internally, `RunTelemetry` delegates to `TelemetryQuery`, which reads the projections and the
artifact store. Coordinates everywhere are `(runId, sequence)` — the same key the log is keyed on.

## Worker wiring

A dedicated telemetry worker-kind (`apps/worker/src/workers/telemetry.ts`) is registered alongside
the sandbox worker in `startWorkers`. It polls `sandbox_runtime_instances` for `running` docker rows
(via a new `SandboxRuntimeInstanceRepo.listRunningDockerInstances`), derives a `SealantTarget` from
each with the existing pure `sealantTargetForRuntimeInstance` seam, and forks one ingester fiber per
run.

- The layer stack is assembled at the boundary; the DB connection config lives only here.
- Self-healing: a failed ingest (for example, the control socket not being up yet) is logged and the
  run is released, so the next poll retries — no explicit readiness probe is needed.
- The whole stack builds into a process-lifetime `Scope`; `stop()` closes it, interrupting every
  fiber and running the epoch-close finalizers.

The launcher was the one decision the design study left open; the recommended "dedicated polling
worker-kind" was chosen for lowest coupling — it survives build-worker restarts and needs no new
queue topology. Switching to a `runtime.launched` RabbitMQ event later is a drop-in.

| scrollback reassembles to `"Hello\n"`                    | byte-exact replay by `streamOffset`        |
| timeline after `rebuild()` equals timeline before        | projections are rebuilds                   |
| `9000000000000000001n` round-trips                       | near-`2^63` bigint fidelity (no narrowing) |

## Caveats & next steps

<Callout type="warn">
  The worker is not yet verified end to end against a booted `sealantd` container. The ingester
  itself is proven against real Postgres with a stub runtime; the worker's live polling → connect →
  ingest path still needs a real sandbox boot to confirm end to end.
</Callout>

### Natural next folds — all additive, no re-capture

- Implement the `reconstruct*` time-travel read models (file tree, process tree, network at a
  `sequence`) — each is a fold of the log up to a ceiling.
- An S3 or filesystem `ArtifactStore` adapter, plus retention and TTL tiering.
- Live tail via Postgres `LISTEN/NOTIFY` or an SSE / tRPC subscription.
- A `ClickHouseTelemetrySink` adapter once raw-log volume justifies it (it slots behind the same
  Tag).
- Poll `HealthReport` into a `telemetry_health_snapshots` table for backpressure-driven
  reconciliation.

### Defaults chosen (all reversible)

Shared control-plane DB, inline-bytea artifacts, keep-forever retention, no `SealantSession` change,
and a dedicated polling telemetry worker. Storage is Postgres for now, behind the pluggable sink.
